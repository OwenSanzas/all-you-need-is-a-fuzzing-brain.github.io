这次跟上次相比有什么变化

上一篇阶段更新(2 月)汇报了 26 个项目里的 62 个漏洞,大多来自对 OSS-Fuzz 已有 harness 的审计,工作面是通用开源软件。从那以后这段工作做的事不一样:重心移到生成新的 harness,目标是一开始就没什么 fuzz 覆盖的项目;目标列表往安全敏感软件再深入了一层。

所有这些工作都跑在一条主线上,我们叫它四原则(Four Principles)。这是四条源码层的检查,每个 harness 在我们认为它产出的 crash 可信之前都得过一遍。简单说就是:(1) harness 自己的代码无 bug;(2) 调用库 API 时遵守库的协议(顺序、生命周期、参数都对);(3) 通过库的公开攻击面驱动它(找出来的 crash 在生产环境里是真能复现的);(4) 选的 fuzz 入口要真的位于一个有意义的攻击面上,而不是某个 no-op helper。第一篇方法论博文里详细介绍了这四条。

三个里程碑、三篇方法论

上面的转向背后是三个连贯的里程碑。先要有一份"什么算好 harness"的可勾选定义;然后用这份定义生成新 harness、找新 bug,以 Chromium 当 showcase;最后给每个候选 crash 上一道 verify 闸门,把误报率压住。每个里程碑对应一篇博文,卡片直接点进去:

2026 年 4 月 30 日 方法论 · Milestone 1

定义"好 harness"是什么

Harness 也会有 bug:基于四原则审计 586 个 OSS-Fuzz Harness。四原则清单(P1 Logic Correctness / P2 API Protocol Compliance / P3 Security Boundary Respect / P4 Entry Point Adequacy),用审计 586 个生产 OSS-Fuzz harness 来验证。提了 53 个 fix PR,45 个被维护者验证或合并,14 个误报在审计阶段就拦住,2 个被失效 harness 长期掩盖的真上游 bug 被翻出来(含 OpenSSL DES 那条潜伏 25 年的)。

2026 年 4 月 30 日 方法论 · Milestone 3

上报前先 verify 每个 crash

我们怎么 verify fuzzer 的 crash:四原则、两次复现、和 Security Proxies。让上线误报率在 5% 以下的 verify 流水线。三道闸:对具体 crash trace 重跑四原则;两次独立复现(harness 重放 + 公开 API repro);跟维护者文档化的 threat model 对齐的 per-library Security Proxies。
