Harness 也会有 bug:基于四原则审计 586 个 OSS-Fuzz Harness

想让 LLM 自动生成"好"的 fuzz harness,先得回答一个更基础的问题:什么样的 harness 才算好,有没有可勾选的源码层定义?这篇提出我们的回答 —— 四原则。要验证它真的对,办法是看它能不能在生产代码上识别坏 harness:我们在 70 个 OSS-Fuzz 项目的 586 个 harness 上跑了一遍,拦下 14 个本会被错报的误报,提了 53 个修复 PR、其中 45 个(85%)被维护者验证或修复,顺带把两个被失效 harness 长期掩盖的真上游 bug 暴露出来 —— 其中一个在 OpenSSL DES 实现里潜伏了 25 年以上。

从一个被绕开的问题说起

"我想让 LLM 自动生成 fuzz harness" —— 这个目标这两年在学术圈和工业界都很热。但深一层看,它依赖一个被普遍跳过的前提:什么样的 harness 才算"好"?没有这个定义,LLM 生成的产物只能用"build 通过"和"60 秒不崩"来打分 —— 这两个信号是必要不充分的,既往工作记录到的误报率高达 94%。

github 上的开源项目正在被大量 AI 工具审计。fuzzing 在这些工具里占有过大份额,因为它自带一个 oracle —— sanitizer。带着 sanitizer trace 的 crash 报告看起来像 ground truth,所以 fuzzing 成了 AI 驱动的 bug 发现和复现的默认载体。问题出在更深的一层。当 harness 自身有 bug,fuzzer 仍然会产出一份"看似合理"的报告:有 sanitizer trace、有看上去合理的调用栈,一应俱全。但那 trace 指向的是任何真实调用者都无法复现的问题。结果是被污染的信号。维护者花精力区分真 bug 和 harness 引发的噪声,而 AI 研究宣称的"自动化"和"规模化",其实悄悄把 triage 负担转嫁给了维护者。

所以核心问题降到了一个 specification 问题。写出一份源码层的、可勾选的"好 harness"定义。

我们的回答是四原则。下一节展开。

光有定义还不够。我们也需要确认这套准则真的能捕捉"好",检验办法很简单。把准则套到一大批生产 harness 上,看它标记的 harness 是否就是真实维护者想修的那些。如果维护者能稳定接受我们提的修复 PR,准则就在追踪一件真东西。如果维护者把 PR 当噪声糊弄掉,这套准则就是噪声。

所以识别坏 harness 不是四原则的下游应用,而是它本身的验证实验。

我们就在 OSS-Fuzz 上跑了这个实验,跨 70 个项目、586 个生产 harness。结果如下。

  • 53 个 violation。每个都做成上游修复 PR。
  • 53 个里 45 个(85%)被维护者 verify 或 merge。35 个已合并;另外 10 个被认定为真问题但 PR 还在排队。
  • 53 个里 14 个本会产生误报级别的 ASan/LSan crash。审计在它们抵达上游之前全部拦下。
  • 修对失效 harness 之后,2 个被长期掩盖的真上游 bug 浮了出来,包括 OpenSSL DES 那条潜伏 25 年的栈缓冲读越界。

后面按这个顺序展开:先讲准则本身,再讲在生产上跑出来的验证结果,最后讲两个被遮住的真上游 bug。


四原则:在源码层定义"正确的 harness"

一个 harness 要被接受,必须同时通过四条源码可检查的条件 —— 我们称之为四原则(Four Principles, P1–P4)。每条原则对应 LLM(或人)写 harness 时最常见的一类失败模式。

P1
Logic Correctness
harness 自身代码无 bug
P2
API Protocol Compliance
调用顺序 / 生命周期 / 参数合法
P3
Security Boundary Respect
鼓励走公开 API,可从公开侧复现
P4
Entry Point Adequacy
目标在真实攻击面上

P1 — Logic Correctness:harness 自己的代码不能有 bug

这是四条里最朴素也最常被违反的一条。harness 是一段 C/C++ 程序,它和任何 C/C++ 程序一样会出错:

  • 变量在使用前先初始化、解引用前先判 NULL。
  • 每条退出路径释放掉所有资源 —— 没有 leak、没有 double-free、harness 内部没有 use-after-free。
  • fuzz 字节真的流到目标 API 的参数里 —— harness 不是 no-op,也没有把 fuzz 输入悄悄换成硬编码常量。
  • LLVMFuzzerTestOneInput 多次调用之间不带遗留的 static / global 状态。
  • API 调用里传的 buffer 大小和 harness 里声明的一致。

P1 违反 ⇒ 根因在 harness 自己,但 sanitizer trace 不一定停在 harness 里。更常见的是 harness 把错误状态(过大 buffer、NULL 指针、未初始化的 struct、被 free 的对象)推给库,然后是库的代码触发 ASan 或 LSan。trace 看起来像库 bug,实际上根因是 caller 违反了协议。这是 P1 类误报里最容易迷惑人的一种。无论 trace 落在哪一边,P1 一旦不通过,这份报告就是噪声。

P2 — API Protocol Compliance:harness 必须遵守库的契约

库的每个 API 都有自己的使用契约:什么时候 init、什么时候 destroy、参数允许哪些范围、返回值必须不必须检查、哪些调用必须配对。LLM 生成的 harness 经常违反这些契约,因为 LLM 没法从单个函数签名看出整个状态机。8 条 P2 子检查在下面 P1.1–P2.8 清单里列全。

举一个真碰到的例子。OSS-Fuzz 上的 libyaml_emitter_fuzzer 想做一件"round-trip 验证"的事:把 fuzz 输入解析成 YAML event,送进 emitter 重新序列化,然后再 parse 一遍序列化结果,看 event 序列是否一致。harness 主控制流(精简后)长这样。 

int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    yaml_parser_t parser;
    yaml_emitter_t emitter;
    yaml_event_t event;
    yaml_event_t events[MAX_EVENTS];
    size_t event_number = 0;

    yaml_parser_initialize(&parser);
    yaml_parser_set_input_string(&parser, data, size);
    yaml_emitter_initialize(&emitter);

    yaml_output_buffer_t out = { NULL, 0, 1000 };
    yaml_emitter_set_output(&emitter, yaml_write_handler, &out);

    /* ---- Loop 1: parse, save event, emit ---- */
    bool done = false;
    while (!done) {
        if (!yaml_parser_parse(&parser, &event)) goto delete_parser;
        done = (event.type == YAML_STREAM_END_EVENT);

        if (copy_event(&events[event_number++], &event)) {  // BUG
            yaml_event_delete(&event);                       //   返回 1 表示成功,
            goto delete_parser;                              //   harness 当成失败处理
        }

        yaml_emitter_emit(&emitter, &event);   // 实际从未被执行
    }

    /* ---- Loop 2: 重新解析输出并比对 event ---- */
    /* (整段都是死代码:Loop 1 永远在 BUG 处退出) */
    yaml_parser_set_input_string(&parser, out.buf, out.size);
    int count = 0;
    while (!done) {
        if (!yaml_parser_parse(&parser, &event)) break;
        if (events_equal(events + count, &event)) break;
        yaml_event_delete(&event);
        count++;
    }

delete_parser:
    yaml_parser_delete(&parser);
    yaml_emitter_delete(&emitter);
    for (int k = 0; k < event_number; k++) yaml_event_delete(events + k);
    free(out.buf);
    return 0;
}

哪错了:第 21 行那一句把判断反了。copy_event() 沿用 libyaml 约定:成功返回 1,失败返回 0。harness 把 1 当成"失败",每次拷成功就直接跳进错误清理 goto。

为什么错:对任何合法 YAML 输入,fuzzer 都会在第一个 event 上短路退出。Loop 1 里的 yaml_emitter_emit 永远不跑。Loop 2 整段(re-parse 加 events_equal 一致性比对)都到不了。整个 harness 大概有 50% 的逻辑从来没被执行过。但 harness 仍能编译、运行不崩、coverage 还在涨(只在那 50% 里),所以在 OSS-Fuzz 上稳稳跑了多年没人发现。

修法只有一个字符:加一个 !。if (copy_event(...)) 改成 if (!copy_event(...))。补丁合并之后,edge coverage 从 166 涨到 2479,+1393%(约 15 倍)。这是教科书级别的 P2.4(Return value handling) 违规,harness 把成功当成失败。完整修复 PR 在 google/oss-fuzz#15096,已合并。

P2 违反 ⇒ 这个 crash 是 API 误用,不是库漏洞。但 P2 还有一种比"上报误报"更隐蔽的失败模式 —— 就是上面这种,harness 编译过、跑起来、coverage 还在涨,却被一个错误的判定悄悄把目标路径清空了。fuzzer 看上去在工作,实际上目标里跟内存安全相关的代码一帧也没走到。你不是没找到 bug,你是根本没在测。这种 case 在 sanitizer 层面没有任何 crash 信号,但整个 fuzz 周期产物为零。比误报还白干。

P3 — Security Boundary Respect:鼓励走公开 API

真实攻击者通常只能从公开导出的 API 进入库。我们鼓励 harness 也尊重这条边界。通过 extern 把 static 函数拉出来直接调,或者跳过公开入口的输入校验直接喂内部 chunk handler,会让 fuzzer 进入没人能从外面到达的代码路径,产生的 crash 难以对应到真实威胁模型。

P3 不是一刀切的"禁止用内部符号"。如果你 100% 确认这个内部用法是合理的(例如该内部状态机的输入语义和某个公开 API 的中间状态完全等价),或者能证明同样的 crash 也可以从公开 API 复现,这种 case 是允许的。判断标准是:能不能用一个只调公开 API 的 harness 把同一个 crash 重现出来 —— 能就 P3 通过,不能就 P3 失败。

  • 鼓励的写法:调公开头文件里声明的函数,不绕过公开入口的输入校验。校验本身是库交付物的一部分。
  • 谨慎使用内部符号:若 harness 必须用 static / friend / 内部 chunk handler,需要在 PR 里明确说明合理性,并附上"同样行为能在公开侧复现"的证据。
  • 反例:直接调 png_handle_IHDR 触发 crash,但任何 PNG 文件都构造不出对应内部状态。这是 P3 失败,因为公开侧无法复现。

P3 是判断题,不是机械题。"某个 static 函数能不能合理拉出来用 / 同一个 crash 公开侧能不能复现"这种问题要靠项目语境 —— 资深 reviewer 读过库的 SECURITY.md、写过类似 fuzzer、了解项目历史,凭经验就能判;对没有项目上下文的第三方通用 LLM 来说不行,它没读过 SECURITY.md、每次给的理由都不一样,容易"看着像内部符号"就一律否掉合理用法。所以 P3(以及下面的 P4)在我们的流水线里都不让 LLM 单点拍板:LLM 列候选,静态调用图给可达性,引用必须落到 file:line,资深 reviewer review borderline。LLM 在这里是工具,不是裁判。

P3 违反 ⇒ 这个 crash 在没人能从公开侧到达的代码路径上。哪怕真崩了也很难修 —— 上游通常会标 NOT_REPRODUCIBLE_FROM_PUBLIC_API。

P4 — Entry Point Adequacy:目标必须在真实攻击面上

P4 决定的是:你选的这个 fuzz 入口,值不值得被 fuzz?一个把所有 fuzz 字节喂给某个 logging helper 或者 trivial accessor 的 harness 即使 P1–P3 全过,也是浪费 CPU。

  • 入口必须真的能 reach 到目标核心实现 —— 不是被早期 return 掉。
  • 入口的可达闭包里必须存在内存安全相关的操作(memcpy、malloc、free、strcpy、sprintf 等),否则即使触发也无害。
  • 目标本身在库的当前公开攻击面上,不是被 deprecated 的 demo 函数。

P4 违反 ⇒ 这个 fuzzer 即使一直跑下去也不会找到有用的 bug。

P1.1–P2.8:16 条可勾选的子检查

把 P1 和 P2 落到工程上,我们各拆成 8 条具体的子检查。审计 / 生成 / verify 三个阶段都用同一套清单,这样三个阶段的判定才能互相对照。

P1 — Logic Correctness 的 8 条子检查

编号 子检查 要求
P1.1资源泄漏每条退出路径都释放掉所有 alloc / fd / lock / handle
P1.2Use-after-free释放之后不再读取或传递该指针
P1.3残留状态多次 LLVMFuzzerTestOneInput 调用之间不带 static / global 状态
P1.4输入流向fuzz 字节真的流到目标 API 的参数里(call 不被常量驱动)
P1.5缓冲安全fuzz buffer 读取有上界、C 字符串带 \0、索引带长度检查
P1.6大小检查输入小于 API 最小值时早返回
P1.7未定义行为没有 OOB / null-deref / 有符号溢出 / 未对齐 cast
P1.8不重新实现调真的库代码,不是 harness 里 inline 的本地复制

P2 — API Protocol Compliance 的 8 条子检查

编号 子检查 要求
P2.1初始化顺序调用目标前所有必需的前置 API 都按顺序到位
P2.2参数构造每个参数的类型 / 范围 / owner / 生命周期都对
P2.3对象生命周期opaque 对象走完 create → configure → use → destroy
P2.4返回值处理检查 return code、走 error branch、对失败 gated 的输出不再使用
P2.5清理顺序所有退出路径上,资源按 API 文档要求的顺序释放
P2.6API 存在性调到的每个函数在 pinned build 的库里真的导出
P2.7共调约束配对 API 共现;互斥 API 不并用
P2.8前置状态外部状态(fd / socket / env / thread)在调目标前就位、之后回收

每条子检查具体怎么落地 —— 哪些用 LLM 读源码 + cite file:line、哪些用 GDB 在目标 API 下断点、哪些用 ASan / LSan 抓 sanitizer trace,以及三个 stage 之间探针输入的差异 —— 我们放到下一篇讲反向流水线时一起展开。这一节先把准则本身列清楚就好。

审计:70 个项目 × 586 个 harness

我们的真正目标是用四原则生成正确的 harness。但要相信它能用在生成阶段,得先证明一件更基础的事:四原则到底能不能识别出 harness 的问题、并指导修复?这一节是这个证明实验。

实验设计是这样:把现有的生产 harness 当作"第一轮 LLM 生成的输出" —— LLM 给一个候选 harness,我们用四原则去审,把找到的问题反馈给 LLM 让它继续改,直到通过。这个 audit-and-refine 的循环就是把"生成"流水线的中间环节单独拎出来,在真实代码上跑给我们看。如果它能在生产代码上稳定地挑出问题、提出能被维护者接受的修复,那放回完整生成流水线里也站得住。

实验对象选 OSS-Fuzz 上 70 个 C/C++ 项目作为基线 —— 这些项目的 harness 大多由项目维护者本人或资深贡献者写,经过上游 PR review,在 ClusterFuzz 上跑了多年。从这 70 个项目里我们抓出了 586 个生产 harness。流程是:

  • 静态过。LLM agent 直接读 harness 源 + 库头文件,跑过 P1.1–P1.8 + P2.1–P2.8 全部 16 个子检查。注意:审计阶段只机械化 P1 和 P2;P3 和 P4 是判断题,由资深 reviewer 单独处理。引用必须落到 file:line。
  • 动态过。对可疑点构造对抗性输入(adversarial probing),跑 60s libFuzzer + ASan / LSan,看违规是否真的触发。
  • 修复并验证。对每个 violation 提交"修好的版本",跑同样 60s 看覆盖率不降。给上游一个直接 merge 的理由。
  • 每个 harness 平均 10 分钟,10 个并发,总成本约 $720。

然后看结果。

53 个 violation,跨 28 个项目

我们一共找到 53 个 P1/P2 violation,占 586 个 harness 的 9.0%,分布在 28 个项目里。剩下的 42 个项目,这一轮没找出违规。需要说明的是,四原则不是确定性判定 —— "没扫出问题"不等于"项目一定干净",只能说在我们这次审计中,这些 harness 没触发任何子检查。这一点边界,值得提醒一下。

53 个全部以 PR / issue 形式提交给了对应上游。截至发稿:

意外战利品:两个被 harness 遮住的真上游 bug

这次审计最值得拿出来讲的不是数字,而是两个本来就在库里、但被错误的 harness 长期遮住的真 bug。修一个 P2 违规,有时会顺手把库的真实行为重新暴露给 sanitizer —— 这正是这次审计里发生的事。

案例 1:OpenSSL DES 栈缓冲读越界 —— 潜伏 25+ 年

OpenSSL 的某个 harness 有一个 API 调用顺序上的 P2 违规,导致整条 cipher 路径在它的执行轨迹里其实是 dead code。修好之后,fuzzer 跑几秒就在 OpenSSL DES 实现里抛出一个 stack-buffer-overread。

这个 bug 的本体在 OpenSSL DES 模块里趟了 25 年以上,比 OpenSSL 这个名字本身还要老。它一直没被发现,不是因为难找,而是过去所有走到这条路径的 harness 都因为同样的 P2 错误根本没真正进到这个函数。一个调用顺序错误,把一个公开 API 的内存安全 bug 隐藏了 1/4 个世纪。

上游 issue:openssl/openssl#30284。修复已合并,issue 已关闭。

案例 2:tidy-html5 内存泄漏 —— 缺一次 cleanup 调用

tidy-html5 的某个 harness 漏掉了一次必要的 cleanup 调用,违反 P2 的"paired API"约束。补上这一次调用之后,harness 立刻在原本被 cleanup 掩盖的路径上抛出 LeakSanitizer 警告 —— 是库内部 lexer 里的真 leak。

根因和上面 OpenSSL 那条完全是一类:错误的 harness 用法把库的真实行为掩盖了。你可以把这两个看作"P2-driven bug discovery"的代表 —— 修 harness 不只是把误报干掉,更是在把库的真实行为重新暴露给 sanitizer。

上游 issue:htacg/tidy-html5#1177。

14 个永远不会变成噪声 bug 报告的误报

53 个 violation 里有 14 个会直接产生误报级别的 ASan/LSan crash。也就是说,如果绕开审计阶段、直接拿这些 harness 去 fuzz,这 14 个 harness 就会冒出"看似真实"的 crash,被自动化流水线打包成 14 份 bug report 砸到上游。

这 14 份报告里没有一个是真 bug。它们的根因都在 harness 里 —— 一个 dict 没 free、一个对象生命周期错位、一个内部 chunk 解析直接被驱动。每一份都会让维护者花时间读 stack trace、构建 repro 环境、对照源码、最后失望地 close 为 NO_VULNERABILITY,顺便给我们的提交账号留下"质量低"的印象。

四原则把这 14 个全部拦在了 fuzz 阶段之前。它们不是"跑出来再被 verify 排除",而是连跑都没跑就被静态识别 + 修复 + 提 PR 的方式消化掉。涉及的项目包括:

  • lcms —— dict 和 transform_ext 两个 harness 都有 P2 违规导致的误报 crash;修完以后干净。
  • libarchive / linkify —— P1 资源生命周期问题。
  • libpng / readapi —— P2 调用顺序错位。
  • njs / script —— P1 静态状态没重置。
  • openvpn —— packet_id、verify_cert 都有 P1 违规导致的假 leak。
  • wamr / mutator —— P2 协议违反。

这是审计的最大杠杆:不让误报流出去,就不需要事后再修复维护者对自动化报告的不信任。

顺带:覆盖率的提升不是目标,但很可观

必须澄清一件事:四原则审计的目的不是涨覆盖率,目的是修正 harness 的正确性。但在很多 case 里,修一个 P1/P2 违规会顺便把死代码激活,覆盖率会跟着涨上去。我们把这当成 PR 通过的"无害证据"附在每个修复里。一些代表性的数字:

  • opencv/filestorage —— +986% edges(原 harness 几乎是死代码)。
  • libyaml/emitter —— 14× edges。
  • openssl/quic_server —— +65.7%。
  • bzip2/bzip2_fd —— +52.4%。
  • tidy-html5/general —— +33.5%。
  • boost/filesystem —— +21.9%。

大幅度提升通常对应"原 harness 是 dead code"的极端 case;小幅度通常是 harness 已经在跑、但有一个非阻塞的 P1/P2 违规(比如 jq/parse_stream)。两种都是 harness 质量问题,只是表现不一样。

为什么这件事在 LLM 时代更重要

手写 harness 的时代,质量问题靠人审 PR 拦掉;LLM 时代,harness 可以一夜之间生成几百个,如果不预设静态质量门槛,自动化就把"speed"翻译成了"误报流量"。

四原则不是"LLM-only"的工具 —— 它是源码层的 harness 正确性定义,既适用于人写的 harness 审计(就是这篇里 586 个的来源),也适用于 LLM 生成 harness 时的自检。我们在生成阶段也用同一套四原则,只是反过来用 —— 那是下一篇的内容。

如果你现在在做 LLM-driven fuzz harness generation,我们建议:

  • 不要把 build 通过和 fuzz 60s 没崩当作"quality good";这两个信号都是必要不充分。
  • 在生成完 harness 之后强制跑 P1.1–P1.8、P2.1–P2.8 的子检查,违规的让 LLM 重写,而不是放进 fuzz queue。
  • P3、P4 在生成之前用,选目标的时候就过一遍;选错了再写代码是浪费。
  • 给上游的每个 PR 必须附 root-cause + 修复 + 覆盖率不降的证据,不能只甩 sanitizer trace。

这套流程的成本是有限的:586 个 harness 平均每个 10 分钟、10 并发,总开销约 $720。换来的是 14 个永远不会成为噪声报告的误报、45 个被维护者验证或修复的 PR(其中 35 个已 merge)、以及两个被失效 harness 长期掩盖的真上游 bug(含 OpenSSL 潜伏 25 年的那条)。这笔账无论从工程化成本还是 ecosystem trust 角度都是正的。
